1Comprar el dominio en NIC Chile

Por qué NIC directo y no GoDaddy: NIC Chile es el registro oficial de .cl; los registradores internacionales solo revenden con sobreprecio (~2×). El dominio se compra una vez y el DNS se delega a donde haga falta — el registrador no importa para el hosting.

  1. Verificar disponibilidad en el whois: https://www.nic.cl/registry/Whois.do?d=sorteatelo.cl
  2. Botón Inscribir → login (o crear cuenta NIC con RUT). ⚠ La sesión de NIC expira rápido (~minutos).
  3. Formulario: titular precargado, facturación Boleta (persona natural), Configuración Técnica se puede dejar vacía (se cambia gratis después).
  4. Crear dominio → carro de compra → plazo 1 año = $9.990 CLP (no hay descuento por más años: 2 años = $19.980 exactos) → pagar con Webpay.
  5. El dominio queda inscrito al confirmarse el pago. La zona .cl publica la delegación en tandas (hasta ~1 hora) — hasta entonces el dominio "no existe" para el resto de internet.
Renovación anual (vence 2027-07-17): si el dominio cae, se caen TODOS los storefronts de los tenants. Dejar recordatorio/renovación con anticipación en nic.cl.

2Crear el proyecto en Vercel y conectar dominios

Por qué Vercel: stack T3/Next.js nativo (deploy automático por push a main), y soporta el requisito duro del ADR-0007: wildcard *.sorteatelo.cl con certificado automático.

  1. Importar el repo GitHub NicoChaima72/sorteatelo como proyecto (framework Next.js autodetectado).
  2. Proyecto → Settings → Domains → Add Existing. Agregar dos dominios apuntando a Production:
    • sorteatelo.cl — el apex = zona de plataforma (landing, panel, onboarding).
    • *.sorteatelo.cl — el wildcard = storefronts de los tenants.
  3. DESMARCAR el checkbox "Redirect apex domains to www": nuestra arquitectura usa el apex como canónico; un www hasta podría confundirse con un slug de tienda en el middleware.
  4. Ambos quedarán "Invalid Configuration" hasta completar el paso 3 — Vercel exige sus nameservers para el wildcard.
Comportamiento esperado post-config: sorteatelo.vercel.app responde 404 a propósito — es un host desconocido para el middleware (ADR-0007: respuesta neutral). Las únicas entradas válidas son sorteatelo.cl y <slug>.sorteatelo.cl.

3Delegar los nameservers a Vercel (en NIC Chile)

Qué significa: Vercel pasa a administrar TODO el DNS del dominio (y así emite el certificado wildcard). Desde este momento, cualquier registro DNS futuro (Resend, correo, verificaciones) se agrega en Vercel, nunca más en NIC.

  1. En clientes.nic.cl → Mis Dominios → sorteatelo.cl → sección 4. Configuración Técnica.
  2. Tipo de servicio: Servidores DNS. Agregar uno por uno:
    • ns1.vercel-dns.com
    • ns2.vercel-dns.com
  3. Actualizar datos de dominio → debe decir "La modificación fue realizada correctamente".
  4. Esperar la tanda de publicación de la zona .cl. Verificar con: 
    nslookup -type=NS sorteatelo.cl a.nic.cl
    Cuando responda los NS de Vercel, en minutos Vercel valida los dominios y emite certificados (el estado en Settings → Domains pasa a "Valid Configuration").

4Base de datos de producción (Supabase)

La app en Vercel no puede ver el Postgres local — necesita uno administrado accesible por internet. Elegido: Supabase (PostgreSQL, free tier, pooler aws-1-us-east-1.pooler.supabase.com).

  1. Crear el proyecto en Supabase y copiar la DATABASE_URL (usar la del pooler en runtime; DIRECT_URL para migraciones de Prisma).
  2. Cargar el schema: con la DATABASE_URL en el .env local, correr 
    npm run db:push   # prisma db push — sincroniza schema.prisma sin migraciones
  3. Verificar que las tablas existen: 
    npx prisma db pull --print | grep "^model "
    Deben aparecer los 15 modelos (Tenant, Product, Order, Raffle, DownloadGrant…).
Deuda anotada en ADR-0015: hoy la MISMA base sirve dev y producción. Antes del go-live público (F10), separar en dos proyectos Supabase (dev / prod) para que el desarrollo no pueda tocar datos reales.

5Variables de entorno en Vercel

Proyecto → Settings → Environment Variables. La lista completa sale de .env.example. Las que dolieron:

VariableValor / origenNotas
DATABASE_URL / DIRECT_URLSupabaseSecretas.
NEXTAUTH_URLhttps://sorteatelo.clEl apex, con https.
NEXTAUTH_SECRETopenssl rand -base64 32Secreta.
GOOGLE_CLIENT_ID / _SECRETGCP → proyecto sortealo → client sortealo-devVer paso 6.
CREDENTIALS_ENCRYPTION_KEYAES-256 base64 (32 bytes)Cifra las FlowCredential (ADR-0006). Perderla = re-cargar credenciales de todos los tenants.
NEXT_PUBLIC_PLATFORM_DOMAINsorteatelo.cl⚠ La que faltó: viene comentada en .env.example y sin ella la app hace fail-fast con 500 en TODA request (ADR-0007). NO es secreta (no marcar "Sensitive").
R2_*Cloudflare R2 (bucket privado de PDFs)Account ID, keys, bucket, endpoint.
R2_PUBLIC_BUCKET / R2_PUBLIC_BASE_URLBucket público de assets (paso 8)No secretas.
RESEND_API_KEYresend.com → API KeysSecreta.
FLOW_URL_CONFIRMATIONhttps://sorteatelo.cl/api/webhooks/flowEn prod ya no hay túnel — es la URL real.

La lección más importante del día: en Vercel las env vars no afectan al deployment que ya corre. Cambiaste una variable ⇒ Redeploy obligatorio (Deployments → menú ⋯ → Redeploy). Las NEXT_PUBLIC_* además se "hornean" en el build del cliente. Nos pasó dos veces: el deploy quedó construido segundos antes de guardar la variable y siguió botado con 500.

6Google OAuth para producción

El client OAuth (sortealo-dev, proyecto GCP sortealo-502704) solo conocía localhost — sin esto el "Entrar con Google" de los Organizadores falla en producción.

  1. console.cloud.google.com/apis/credentials → ojo con estar en la cuenta y proyecto correctos (el ID del client en el .env empieza con el número del proyecto: 274688118230-…).
  2. Editar el client → Orígenes autorizados de JavaScript → agregar https://sorteatelo.cl.
  3. URIs de redireccionamiento autorizados → agregar https://sorteatelo.cl/api/auth/callback/google (el path exacto de NextAuth).
  4. Guardar. Google avisa: puede tardar de 5 minutos a horas en aplicar.
Para el go-live serio conviene un client de producción separado (y publicar la pantalla de consentimiento fuera de "Testing" para que cualquier Organizador pueda loguearse).

7Resend: verificar el dominio remitente

Para enviar como @sorteatelo.cl (y no onboarding@resend.dev) Resend exige probar la propiedad del dominio vía DNS. No se necesita contratar ningún buzón para enviar.

  1. resend.com/domains → Add domain → sorteatelo.cl, región São Paulo (sa-east-1).
  2. Resend entrega 3-4 registros. Se agregan en Vercel (Team → Domains → sorteatelo.cl → DNS Records):
    TipoNameValuePrio
    TXTresend._domainkeyp=MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKB… (la clave DKIM que muestra Resend)—
    MXsendfeedback-smtp.sa-east-1.amazonses.com10
    TXTsendv=spf1 include:amazonses.com ~all—
    TXT_dmarcv=DMARC1; p=none; (opcional, recomendado)—
  3. En Resend: I've added the records → estado Pending / Checking DNS → pasa a Verified solo (requiere que la zona .cl ya esté publicada).
  4. Al verificarse, cambiar el remitente en el código: la constante REMITENTE_CORREO en src/server/domain/correo/plantillaDescarga.ts (D6: un solo lugar) — de onboarding@resend.dev a algo como no-responder@sorteatelo.cl. La marca visible ya es MARCA_PLATAFORMA = "Sortéatelo".
Recibir correo en contacto@sorteatelo.cl es otra cosa: se resuelve gratis con Cloudflare Email Routing o Zoho Mail free cuando se necesite (los MX del apex están libres).

8Bucket R2 público de assets (ADR-0013)

Los assets de marca de las tiendas (logo, hero, portadas, premio) van en un segundo bucket con lectura pública — separado a nivel de bucket del privado de PDFs (sortealo-dev), para que un PDF jamás pueda filtrarse por URL pública. Reusa las mismas credenciales R2 de la cuenta.

  1. Cloudflare Dashboard → R2 → Create bucket → nombre sortealo-public-dev (consistente con el naming dev existente).
  2. Bucket → Settings → Public access → R2.dev subdomain → Allow. Copiar la URL https://pub-<hash>.r2.dev.
  3. Verificar que el API token R2 existente tenga permiso Object Read & Write sobre este bucket (si estaba scopeado solo a sortealo-dev, editar el token y sumar el bucket nuevo — no hace falta regenerar las keys).
  4. Env vars (local y Vercel, no secretas): 
    R2_PUBLIC_BUCKET="sortealo-public-dev"
R2_PUBLIC_BASE_URL="https://pub-<hash>.r2.dev"   # SIN barra final
  5. Redeploy (regla del paso 5). Mientras no existan, la plantilla degrada elegante (gradientes en vez de imágenes) — no rompe nada.

9Cómo verificar que todo quedó bien

# 1. La zona .cl ya delegó a Vercel?
nslookup -type=NS sorteatelo.cl a.nic.cl        # → ns1/ns2.vercel-dns.com

# 2. El apex responde? (200 = plataforma arriba)
curl -sS -o /dev/null -w "%{http_code}\n" https://sorteatelo.cl

# 3. Un storefront de tienda responde?
curl -sS -o /dev/null -w "%{http_code}\n" https://<slug>.sorteatelo.cl

# 4. El host neutro responde 404 (correcto, ADR-0007)
curl -sS -o /dev/null -w "%{http_code}\n" https://sorteatelo.vercel.app
  • Errores de runtime: Vercel → proyecto → Logs (o la herramienta get_runtime_logs). Un 500 en toda ruta con "Falta NEXT_PUBLIC_PLATFORM_DOMAIN" = env var faltante + redeploy.
  • Login Google: probar "Entrar con Google" en https://sorteatelo.cl/login. Error redirect_uri_mismatch = paso 6 incompleto o aún propagando.
  • Resend: dashboard → Domains → sorteatelo.cl = Verified; enviar un correo de prueba.
